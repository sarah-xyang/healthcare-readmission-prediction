## 1. Project Business Context

**What this project does:** Predicts which hospital patients are at high risk of being readmitted within 30 days of discharge.

**Why it matters:** The Centers for Medicare & Medicaid Services (CMS) penalizes hospitals under the Hospital Readmissions Reduction Program (HRRP) when too many patients return within 30 days. These penalties can reach 3% of total Medicare payments which is a significant financial hit for most hospitals. By identifying high-risk patients *before* discharge, care teams can intervene: scheduling follow-up calls, arranging home health visits, or extending observation stays.

**Who uses the output:** Hospital administrators and care coordinators, not data scientists. They need clear, actionable outputs — not raw model scores.

**Definition of success:** A model that correctly flags a meaningful portion of true readmissions while keeping the false-positive rate low enough that care teams aren't overwhelmed with low-risk patients.

## 2. Working with the Project Owner

**Background:** The project owner has a strong product management background and is learning data science. They understand business problems deeply but may be less familiar with technical data science conventions.

**How to write code and comments:**
- Every function must have a docstring that starts with the *business purpose* — what decision or outcome does this function support? — before explaining the technical mechanics.
- Do not use unexplained acronyms or jargon (e.g., write "AUC (a measure of how well the model separates readmitted from non-readmitted patients)" not just "AUC").
- When making a modeling choice (e.g., choosing XGBoost over logistic regression), briefly explain the business trade-off, not just the technical reason.
- Notebook markdown cells should explain what we are about to do and *why it matters for the business problem*, not just label the code block.

## 3. Technical Standards

### Python Style
- Follow PEP 8 (standard Python style guide). Line length limit: 88 characters (Black formatter default).
- Use descriptive variable names that hint at the business meaning: `high_risk_patients` not `df_filtered`, `penalty_savings_usd` not `val`.
- Avoid abbreviations unless they are universally understood (e.g., `df` for DataFrame is fine).

### Docstrings
Use Google-style docstrings:

```python
def calculate_penalty_savings(predictions, avg_penalty_per_readmission):
    """Estimate the annual Medicare penalty savings from model-driven interventions.

    Hospitals pay CMS penalties proportional to their excess readmission ratio.
    This function converts model predictions into a dollar estimate of how much
    those penalties could be reduced if flagged patients receive follow-up care.

    Args:
        predictions (pd.Series): Binary predictions (1 = high risk, 0 = low risk).
        avg_penalty_per_readmission (float): Estimated CMS penalty cost per avoided
            readmission, in USD.

    Returns:
        float: Estimated annual penalty savings in USD.
    """
```

### Notebooks
- Keep individual code cells focused: one logical step per cell.
- Every section must start with a markdown cell explaining the business purpose of that section.
- Do not leave dead code or unused imports in notebooks.
- Notebook output (charts, tables) should be cleared before committing — outputs are regenerated by running the notebook.

### Commit Messages
Use this format:
```
<type>: <short description>

<optional body explaining the business reason for the change>
```
Types: `feat` (new feature), `fix` (bug fix), `data` (data changes), `docs` (documentation), `refactor` (code restructure, no behavior change).

Example:
```
feat: add 6-month readmission history feature

Prior admissions are one of the strongest predictors of future readmission.
This feature counts how many times each patient was admitted in the 6 months
before their current discharge date.
```

## 5. Key Workflows

### Adding a new predictive feature
1. Implement the feature engineering logic in `src/features.py` with a Google-style docstring explaining the clinical or business rationale.
2. Add the feature to the preprocessing pipeline in `notebooks/02_preprocessing.ipynb`.
3. Re-run `notebooks/03_model.ipynb` to evaluate whether the feature improves model performance.
4. If it helps, document it in the "Features" section of `README.md`.

### Retraining the model
1. Run `notebooks/02_preprocessing.ipynb` to regenerate `data/processed/`.
2. Run `notebooks/03_model.ipynb` — the trained model is saved to `outputs/models/`.
3. Use MLflow (already in `requirements.txt`) to log the experiment run with parameters and metrics.
4. Run `notebooks/04_business_impact.ipynb` to update business impact estimates.

### Updating business impact calculations
1. Edit functions in `src/business_metrics.py`.
2. Re-run `notebooks/04_business_impact.ipynb` to regenerate outputs.
3. Update the "Results" section of `README.md` with new figures.

## 6. Do Not

- **Do not modify files in `data/raw/`** — ever. Raw data is immutable. If you need to change how the data is processed, change `src/data_processing.py` or `notebooks/02_preprocessing.ipynb`, not the source files.
- **Do not skip business-context comments.** Every function and every notebook section must explain its business purpose, not just what the code does technically.
- **Do not add dependencies** not in `requirements.txt` without discussing it first. Dependency changes can break reproducibility.
- **Do not commit notebook outputs** (charts, tables, model outputs inside `.ipynb` files). Clear outputs before committing.
- **Do not use overly terse variable names** (`x`, `tmp`, `df2`) — use names that reflect the business domain.
- **Do not hardcode file paths** — use `pathlib.Path` and define paths relative to the project root.
